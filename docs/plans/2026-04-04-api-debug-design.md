# API 调试页重构设计

## 背景

当前 App 内的“API 调试”页面仍然是通用请求器思路：

- 手动选择方法
- 手动输入路径
- 手动拼 Query
- 手动输入 JSON Body

这虽然自由，但不符合当前项目的真实使用场景。用户真正需要的是：

- 按核心前端已有能力快速验证关键接口
- 用手机端顺手地完成自动匹配、手动匹配与弹幕拉取
- 保持与当前 App 首页、核心页、记录页一致的界面风格

因此这次重构不再保留“万能请求器”，改为核心前端思路的 App 化版本。

## 目标

把“API 调试”页重构为双主路径工具页：

1. 接口调试
2. 弹幕测试

同时满足：

- 只保留高价值接口
- 以移动端交互为主，不照搬网页布局
- 操作流畅，状态清晰
- 不新增后端接口

## 设计原则

1. 只做真实需要的调试能力，不保留泛用低频入口
2. 使用预设接口替代自由拼接
3. 弹幕测试优先走流程，不走“把所有区块都摊开”的网页方式
4. 结果区高信息密度，但层级明确
5. 视觉语言继续沿用当前沉浸式 App 风格

## 页面结构

页面采用两级结构：

### 一级切换

- 接口调试
- 弹幕测试

一级切换使用紧凑的页内分段切换器，不再做成长列表入口。

## 接口调试

### 定位

“接口调试”不是任意请求器，而是预设接口测试台。

### 支持的接口

保留核心前端中的高价值接口：

- 搜索动漫 `/api/v2/search/anime`
- 搜索剧集 `/api/v2/search/episodes`
- 匹配动漫 `/api/v2/match`
- 获取番剧详情 `/api/v2/bangumi/:animeId`
- 获取弹幕 `/api/v2/comment/:commentId`
- 获取分片弹幕 `/api/v2/segmentcomment`

### 结构

1. 接口选择区
   - 选择预设接口
   - 显示方法与路径摘要

2. 参数表单区
   - 根据接口定义动态渲染字段
   - 支持文本、下拉、路径参数、JSON Body
   - 仅显示当前接口需要的字段

3. 发送动作区
   - 只有一个主按钮：发送请求
   - 支持回车直接触发

4. 结果工作台
   - 请求摘要
   - 响应状态
   - 响应正文

### 交互要求

- 切换接口时表单平滑替换
- 请求中保持上一次结果，避免整块闪空
- 必填字段在本地先校验
- 错误直接收敛到结果区，不额外制造多余提示层

## 弹幕测试

### 顶层结构

“弹幕测试”内部再分两条链路：

- 自动匹配
- 手动匹配

### 自动匹配

#### 输入

- 文件名

#### 执行链路

- 调用匹配接口
- 取最佳结果
- 自动获取弹幕

#### 结果区

统一收口为一个工作台，展示：

- 匹配结果摘要
- 命中标题
- 关键 ID
- 弹幕统计
- 弹幕预览

### 手动匹配

#### 交互方式

不采用“下面展开越来越长”的网页结构，而采用页内步骤切换。

步骤如下：

1. 搜索
2. 番剧
3. 剧集
4. 结果

#### 交互规则

- 当前步骤只显示当前内容
- 上一步数据保留在状态中，但不继续占屏
- 每一步都有明确返回入口
- 当前所处步骤有清晰标识

#### 各步骤职责

1. 搜索
   - 输入关键字
   - 发起搜索动漫请求

2. 番剧
   - 展示搜索结果列表
   - 选择目标番剧

3. 剧集
   - 展示番剧剧集列表
   - 选择目标剧集并获取弹幕

4. 结果
   - 展示弹幕统计
   - 展示弹幕预览
   - 提供导出或查看完整响应入口

## 视觉方向

本页继续沿用当前管理器重构版风格：

- 沉浸背景
- 紧凑卡片
- 小而明确的切换器
- 高信息密度结果区

避免出现：

- 网页后台式大段说明
- 多个无关卡片连续堆叠
- 过多状态标签
- 过厚的表单边框

## 数据与实现边界

- 不新增后端接口
- 继续通过现有 `requestDanmuApi()` 完成所有请求
- App 侧新增预设接口配置
- App 侧新增弹幕测试状态与响应解析逻辑

## 验收标准

重构完成后应满足：

1. 页面顶层明确分为“接口调试”和“弹幕测试”
2. 不再保留手填任意路径的万能请求器
3. 接口调试使用预设接口并动态生成参数表单
4. 弹幕测试包含自动匹配与手动匹配两条链路
5. 手动匹配采用页内步骤切换，不做纵向展开堆叠
6. 结果展示与现有 App 风格统一，信息密度高且层级清晰
