# Danmu Manager Redesign V2 Design

## 背景

当前重构版已经完成 Compose + Material 3 基础重写，但整体界面仍然偏“堆料式控制台”：

- 单个页面同时承载状态展示、操作入口、配置项和大段说明
- 一级视觉重点过多，按钮密度偏高
- 日志、请求、调试、设置等高级能力与日常主任务同权
- 手机端滚动过长，用户进入页面后难以快速判断“现在该做什么”

本轮 V2 设计不再沿用现有布局，只保留现有功能与数据接线，重新定义导航、信息架构和视觉优先级。

## 设计目标

1. 首屏 3 秒内能看懂服务是否健康、当前核心是谁、下一步该做什么
2. 一级页面只承载一个核心问题，不再把监控、配置、说明混在一起
3. 高级能力默认后置，主路径只保留高频操作
4. 手机端优先，平板/横屏在此基础上增强为双栏而不是堆更多卡片
5. 保持全部现有功能可达，不删能力，只重组入口

## 体验方向

整体风格改为“极简运维台”，采用“极简控制台骨架 + 少量仪表盘信息”的折中方案。

视觉关键词：

- 冷静
- 清楚
- 偏系统工具感
- 低认知负担

不再强调“命令中心”式大 Hero、大量状态胶囊和高频亮色按钮，而改为：

- 更平静的浅色底
- 更强的标题层级
- 更少的同时可见操作
- 更明确的状态分区

## 信息架构

一级导航调整为 4 个页面：

1. `主页`
2. `核心`
3. `监控`
4. `设置`

导航规则：

- 手机端使用 `NavigationBar`
- 大屏使用 `NavigationRail`
- `核心` 与 `设置` 在大屏上使用 list-detail 结构

## 页面设计

### 1. 主页

主页只回答一个问题：`现在系统状态如何，我最该做什么？`

页面结构：

```text
[状态总条]
运行状态 / Root / 自启动 / 当前核心

[主操作区]
启动 or 停止
重启
刷新

[实例摘要]
模块版本
核心版本
今日请求
接口地址

[最近提醒]
最多 3 条
每条带一个动作

[快捷任务]
检查核心更新
检查模块更新
导出配置

[访问入口]
默认折叠
展开后显示 Token / URL / 局域网地址
```

关键原则：

- 删除大段介绍文本
- 删除当前大面积 Hero Card 视觉占比
- 摘要指标限制在 4 项以内
- URL 与 Token 默认折叠，避免首屏被代码块占满

### 2. 核心

核心页只回答一个问题：`我现在有哪些核心，它们的状态如何，我要切换还是安装？`

页面拆成两层：

- 默认层：已安装核心列表
- 次级层：安装新核心

手机端结构：

```text
[搜索与过滤]

[核心列表]
名称
ref
版本
状态 badge

[详情底部抽屉/独立卡片]
版本
commit
安装时间
大小
更新状态
操作按钮

[安装核心入口]
右下角 FAB / 顶部按钮
点开 bottom sheet
```

大屏结构：

- 左侧列表
- 右侧详情

关键原则：

- 已安装资源优先于安装表单
- 安装流程不长期占据首屏
- 切换/更新/删除收束到详情区，不在列表中排一整排按钮

### 3. 监控

监控页只回答一个问题：`最近发生了什么，系统是否异常？`

将原 `Console` 重组为 4 个二级分段：

1. `事件`
2. `日志`
3. `请求`
4. `调试`

#### 事件

新增统一事件时间线，汇总：

- 服务启动/停止/重启
- 核心安装/切换/删除
- 更新检查
- 关键错误

#### 日志

只保留：

- 日志文件切换
- 刷新
- 清空
- 正文查看

#### 请求

列表化展示：

- method
- path
- 时间
- 客户端

#### 调试

保留 API 调试器，但降级为高级能力：

- 默认不做首标签
- 结构简化
- 说明收短

关键原则：

- 首默认标签改为“事件”，而不是原始日志
- API 调试不再与日志/请求同等抢注意力
- 一个子页只做一件事

### 4. 设置

设置页只回答一个问题：`我需要改哪一类配置？`

改为分类入口页，再进入详情页：

1. `访问与令牌`
2. `备份与恢复`
3. `显示与主题`
4. `日志与维护`
5. `高级设置`

结构：

```text
[设置分类列表]

点击分类后进入详情页：
- 当前分类对应表单
- 底部保存区
- 危险操作区固定在末尾
```

额外规则：

- `.env` 编辑器单独做全屏页
- WebDAV 不再与主题、日志参数混排
- 风险操作统一放入危险分区

## 视觉系统调整

### 色彩

- 默认主背景改为暖灰白，而不是大纯白
- 主强调色保留一组冷蓝青色
- 成功使用偏绿色状态色
- 警告使用琥珀色
- 错误使用偏红色
- 动态配色仅作为辅助，不主导整体观感

### 组件

- 减少高饱和 Filled Tonal Button 的数量
- 主按钮统一尺寸与位置
- 以较轻卡片 + 分组标题替代大型 Hero 面板
- badge 统一状态语义，不再混用 pill/tag/chip 语气

### 排版

- 页面标题更短
- 说明文字减少 40% 以上
- 一屏只允许一个一级视觉重点
- 所有设置说明尽量压成一句话

## 响应式规则

### 手机

- 单列滚动
- 折叠高级内容
- 重要操作放在可见区域上半部

### 平板 / 横屏

- 核心页与设置页使用双栏
- 监控页保留单主栏，避免左右分裂太强
- 主页允许摘要区两列，但仍只保留一个主操作区

## 功能映射

V2 不删除现有功能，只重组到新入口：

- 服务启停、自启动：主页
- 核心安装/切换/更新/删除：核心
- 服务日志、模块日志、请求记录、API 调试：监控
- 主题、动态配色、GitHub Token、WebDAV、日志清理、console 行数、`.env` 导入导出：设置

## 实施策略

1. 先重写导航壳与一级页面结构
2. 再重构主页和核心页，建立新的信息优先级
3. 再重构监控页和设置页，拆掉长滚动混排
4. 最后统一设计系统组件，清理旧 Hero / Metric / 说明性样式

## 验收标准

1. 手机端首页首屏只出现一个主操作区
2. `核心` 页面默认先看到已安装核心，而不是安装表单
3. `监控` 默认首屏是事件/摘要，不是长日志
4. `设置` 不再是单页大杂烩长表单
5. 保持现有功能完整可达
6. `:app:assembleDebug` 可通过
